Introdução

O que você vai construir

Este tutorial vai guiar você na criação de um sistema de agendamento completo, rodando no seu próprio computador, usando o Claude Code como ferramenta de desenvolvimento.

O sistema que você vai construir ao longo deste tutorial faz o seguinte: cadastra clientes com nome e telefone, cadastra serviços com nome e duração, cria agendamentos associando cliente, serviço, data e horário, mostra a agenda do dia e da semana em ordem cronológica, e permite editar ou cancelar agendamentos existentes. Todos os dados ficam salvos em um arquivo local no seu computador.

Por que local e por que SQLite

O sistema roda inteiramente no seu computador. Não precisa de internet para funcionar, não depende de servidores externos e não tem custo mensal de hospedagem. Isso significa que você tem controle total sobre seus dados e que o sistema funciona mesmo sem conexão.

O banco de dados usado é o SQLite, que armazena tudo em um único arquivo. Esse arquivo pode ser copiado como backup, movido para outro computador ou guardado em um pendrive. Sem configuração de servidor de banco de dados, sem senhas de acesso, sem complexidade desnecessária.

O que está fora do escopo

Este tutorial não cobre: envio de notificações para clientes, acesso pela internet, controle de login e senhas, relatórios financeiros ou integração com outros sistemas. Essas funcionalidades adicionam complexidade significativa e ficam para um momento posterior, quando o sistema básico já estiver funcionando.

Pré-requisitos

Para seguir este tutorial, você precisa ter instalado no seu computador: Node.js (versão 18 ou superior), o Claude Code (instalado via npm), e um navegador web qualquer. Se você ainda não tem o Claude Code instalado, o Capítulo 1 cobre esse processo.

Dica: siga na ordem

Cada capítulo depende do anterior. Não pule etapas. Se algo não funcionar em um capítulo, resolva antes de avançar para o próximo.

Preparando o ambiente

Antes de pedir qualquer coisa ao Claude Code, você precisa garantir que o ambiente está configurado corretamente. Esse capítulo cobre a verificação da instalação, a criação da pasta do projeto e a configuração inicial.

Verificando se o Claude Code está instalado

Abra o terminal do seu computador. No Windows, procure por "Terminal" ou "Prompt de Comando" no menu iniciar. No macOS, abra o aplicativo "Terminal" na pasta Utilitários. Digite o seguinte comando e pressione Enter:

Terminal 
claude --version

Se aparecer um número de versão (algo como 1.0.x), o Claude Code está instalado. Se aparecer uma mensagem de erro dizendo que o comando não foi encontrado, você precisa instalá-lo primeiro:

Terminal 
npm install -g @anthropic-ai/claude-code

Se o comando npm também não for reconhecido, significa que o Node.js não está instalado. Acesse nodejs.org, baixe a versão LTS e instale. Depois, feche e reabra o terminal antes de tentar novamente.

Criando a pasta do projeto

Escolha um local no seu computador para guardar o projeto. Pode ser a área de trabalho, a pasta Documentos ou qualquer outro lugar que faça sentido para você. Crie uma pasta chamada agendamento:

Terminal 
mkdir agendamento
cd agendamento

A partir de agora, todos os comandos devem ser executados dentro dessa pasta. Sempre que abrir o terminal para trabalhar no projeto, navegue até ela primeiro.

Iniciando o Claude Code dentro do projeto

Com o terminal aberto na pasta agendamento, inicie o Claude Code:

Terminal 
claude

O Claude Code vai abrir e estará pronto para receber instruções. Ele já sabe que está dentro da pasta agendamento e vai criar todos os arquivos do projeto ali.

Criando o arquivo CLAUDE.md

O primeiro passo dentro do Claude Code é criar o arquivo CLAUDE.md. Esse arquivo funciona como um manual de instruções para o próprio Claude Code. Nele, você descreve o que o projeto faz, quais tecnologias usa e quais regras o agente deve seguir.

Digite a seguinte instrução no Claude Code:

Instrução para o Claude Code 
Crie um arquivo CLAUDE.md na raiz do projeto com as seguintes informações:

- Tecnologias: Node.js, Express, SQLite, HTML/CSS/JavaScript puro

- O banco de dados é um arquivo SQLite local

O que é o CLAUDE.md e por que ele importa

O CLAUDE.md é o arquivo que o Claude Code lê automaticamente toda vez que você inicia uma sessão dentro de um projeto. Ele dá contexto ao agente sobre o que está sendo construído, evitando que você precise repetir informações básicas a cada nova conversa.

Sem esse arquivo, o Claude Code não tem como saber que tipo de projeto você está construindo. Com ele, o agente já começa cada sessão sabendo que se trata de um sistema de agendamento local, que usa SQLite, que roda em Node.js e que a interface é acessada pelo navegador.

Ao longo do tutorial, você vai adicionar mais informações a esse arquivo conforme o projeto evolui. Por enquanto, o conteúdo básico é suficiente.

Inicializando o Git

Antes de começar a escrever código, inicialize o controle de versão. O Git permite que você volte a versões anteriores do projeto caso algo quebre. Peça ao Claude Code:

Instrução para o Claude Code 
Inicialize um repositório Git neste projeto e faça o primeiro commit
com a mensagem "setup inicial do projeto".

A partir daqui, sempre que uma funcionalidade estiver funcionando, peça ao Claude Code para fazer um commit. Isso cria pontos de restauração que você pode usar caso precise voltar atrás.

Atenção: não pule o Git

O Git é sua rede de segurança. Sem ele, se algo quebrar de forma grave, você pode perder todo o trabalho feito até aquele ponto. Com ele, você sempre pode voltar ao último estado funcional.

Definindo o sistema antes de escrever uma linha

O erro mais comum ao usar Claude Code é começar a pedir funcionalidades sem ter definido o que o sistema precisa fazer. Isso gera retrabalho, inconsistências e frustrações. Definir antes de construir economiza tempo.

Por que planejar antes de pedir código

O Claude Code executa o que você pede. Se você pedir coisas vagas, o resultado será vago. Se você pedir coisas contraditórias, o resultado será inconsistente. A solução é simples: antes de pedir qualquer código, defina com clareza o que o sistema precisa fazer.

Planejar não significa escrever um documento de 50 páginas. Significa listar as funcionalidades mínimas que o sistema precisa ter para ser útil. Nada mais, nada menos.

Lista mínima de funcionalidades

Para um sistema de agendamento funcional, você precisa de:

Funcionalidades essenciais
Cadastro de clientes: nome e telefone. Poder listar, editar e excluir clientes.
Cadastro de serviços: nome do serviço e duração em minutos.
Criação de agendamentos: associar um cliente a um serviço, com data e horário definidos.
Visualização da agenda do dia: ver todos os agendamentos de um dia específico, em ordem cronológica.
Visualização da agenda da semana: ver os agendamentos da semana inteira de forma resumida.
Edição e cancelamento: poder alterar data, horário ou serviço de um agendamento, ou cancelá-lo.

Essa lista é o escopo do projeto. Tudo que está nela será construído. Tudo que não está nela fica de fora, pelo menos por enquanto.

O conceito de escopo

Escopo é o limite do que o sistema faz. Definir escopo significa dizer "o sistema faz isso e não faz aquilo". Sem escopo definido, você corre o risco de ficar adicionando funcionalidades indefinidamente sem nunca ter um sistema funcional.

Para este projeto, o escopo é claro: um sistema local de agendamento para uso pessoal do dono do negócio. Não é um sistema online, não tem login de clientes, não envia mensagens automáticas. Essas decisões não são limitações; são escolhas que mantêm o projeto viável.

A primeira instrução real ao Claude Code

Com o escopo definido e documentado, você pode dar a primeira instrução de desenvolvimento ao Claude Code. Mas ainda não é hora de pedir código. A primeira instrução deve ser para o agente criar a estrutura de pastas do projeto:

Instrução para o Claude Code 
Crie a estrutura de pastas para o projeto de agendamento:
- src/ para o código do servidor
- public/ para os arquivos da interface (HTML, CSS, JS)
- database/ para o arquivo do banco de dados
Não crie nenhum código ainda, apenas as pastas e arquivos vazios
com comentários indicando o que cada um vai conter.

Essa abordagem, criar a estrutura antes do conteúdo, ajuda o Claude Code a organizar o código de forma consistente nas etapas seguintes. Quando você pedir para criar o servidor, ele já sabe que o arquivo vai em src/. Quando pedir a interface, ele sabe que vai em public/.

Dica: instruções específicas geram resultados melhores

Compare "crie um sistema de agendamento" com a instrução acima. A primeira é vaga e pode gerar qualquer coisa. A segunda é precisa e gera exatamente o que você precisa. Quanto mais específica a instrução, melhor o resultado.

"Defina o que você quer antes de pedir. O Claude Code executa instruções, não adivinha intenções."

Skills que você vai precisar

Skills são extensões que adicionam conhecimento especializado ao Claude Code. Sem elas, o agente funciona com seu conhecimento geral. Com elas, ele segue padrões específicos e produz resultados mais consistentes.

O que são skills

Pense em skills como manuais de referência que o Claude Code consulta quando precisa executar um tipo específico de tarefa. Uma skill de frontend, por exemplo, contém padrões de design, boas práticas de interface e estruturas de código que o agente usa como base ao criar telas.

Skills não mudam o que o Claude Code pode fazer. Elas mudam a qualidade e a consistência do que ele faz. Um agente sem skills pode criar uma interface funcional, mas com a skill de frontend instalada, ele cria interfaces mais organizadas, com melhor estrutura visual e código mais limpo.

Skills recomendadas para este projeto

Para o sistema de agendamento, quatro skills são particularmente úteis:

Skills recomendadas
frontend-design: melhora a qualidade visual das telas criadas. Gera interfaces com design mais profissional e código CSS mais organizado.
react-nextjs-development: embora o projeto use HTML puro, essa skill contém padrões de organização de componentes que o agente aplica mesmo em projetos sem React.
testing-debugging: ajuda o agente a diagnosticar e corrigir erros de forma sistemática. Essencial quando algo não funciona como esperado.
simplify: revisa o código gerado e sugere simplificações. Útil para manter o projeto limpo e fácil de entender.

Como instalar cada skill

A instalação de skills é feita dentro do próprio Claude Code. Para cada skill, use o comando /install-skill seguido do nome:

Dentro do Claude Code 
/install-skill frontend-design
/install-skill react-nextjs-development
/install-skill testing-debugging
/install-skill simplify

Cada skill é instalada uma única vez. Depois de instalada, ela fica disponível em todas as sessões futuras do Claude Code naquele projeto. Você não precisa reinstalar a cada vez que abrir o agente.

Quando usar cada skill

Você não precisa ativar skills manualmente na maioria dos casos. O Claude Code identifica automaticamente quando uma skill é relevante para a tarefa que você pediu. Porém, em situações específicas, você pode acionar uma skill diretamente:

Quando acionar cada skill
frontend-design: acione quando pedir para criar ou modificar telas. Use /skill frontend-design antes de descrever o que você quer na interface.
testing-debugging: acione quando algo não funciona. Use /skill testing-debugging antes de descrever o erro que você encontrou.
simplify: acione depois que uma funcionalidade estiver pronta. Use /skill simplify para pedir uma revisão do código gerado.

O que acontece sem skills

O Claude Code funciona sem nenhuma skill instalada. A diferença é na qualidade e consistência dos resultados. Sem a skill de frontend, por exemplo, o agente ainda cria interfaces funcionais, mas pode usar estilos inconsistentes entre telas diferentes ou gerar código CSS desorganizado.

Sem a skill de testing-debugging, o agente ainda tenta corrigir erros, mas pode não seguir um processo sistemático de diagnóstico. Ele pode tentar soluções aleatórias em vez de investigar a causa raiz do problema.

As skills não são obrigatórias, mas são recomendadas. Elas reduzem a quantidade de ajustes que você precisa pedir depois e aumentam a chance de o resultado ficar correto na primeira tentativa.

Dica: instale as skills antes de começar a construir

O melhor momento para instalar skills é agora, antes de pedir qualquer código. Assim, todo o código gerado a partir do próximo capítulo já se beneficia das skills instaladas.

Atenção: skills não substituem instruções claras

Skills melhoram a qualidade do resultado, mas não compensam instruções vagas. Mesmo com todas as skills instaladas, você ainda precisa descrever com precisão o que quer. A combinação de skills boas com instruções claras é o que gera os melhores resultados.

Configurando o banco de dados SQLite

O banco de dados é onde o sistema guarda todas as informações: clientes, serviços e agendamentos. Sem ele, os dados desapareceriam toda vez que você fechasse o sistema.

O que é SQLite, em linguagem simples

Um banco de dados é como uma planilha organizada que o computador consegue consultar rapidamente. O SQLite é um tipo específico de banco de dados que guarda tudo em um único arquivo no seu computador. Não precisa de instalação separada, não precisa de configuração de servidor, não precisa de senha.

Quando você cadastra um cliente no sistema, o SQLite grava essa informação no arquivo. Quando você abre a lista de clientes, o SQLite lê o arquivo e retorna os dados. Tudo acontece localmente, sem internet.

Por que SQLite é a escolha certa para este projeto

Existem outros bancos de dados mais conhecidos, como MySQL ou PostgreSQL. Eles são poderosos, mas exigem instalação e configuração de um servidor separado. Para um sistema local usado por uma pessoa, isso é complexidade desnecessária.

O SQLite é suficiente para armazenar milhares de agendamentos sem nenhum problema de desempenho. Ele só se torna limitado quando muitas pessoas precisam acessar o banco ao mesmo tempo, o que não é o caso de um sistema local.

Como pedir ao Claude Code para configurar o banco

Com o Claude Code aberto na pasta do projeto, dê a seguinte instrução:

Instrução para o Claude Code 
Configure o banco de dados SQLite para o sistema de agendamento.
Crie as seguintes tabelas:

1. clientes: id, nome, telefone, observações, data_criacao
2. servicos: id, nome, duracao_minutos, data_criacao
3. agendamentos: id, cliente_id (referência clientes), servico_id
   (referência serviços), data, horario, status (agendado/cancelado),
   observações, data_criacao

Use o pacote better-sqlite3. O arquivo do banco deve ficar em
database/agendamento.db. Crie um arquivo src/database.js que
inicializa o banco e cria as tabelas se elas não existirem.

O que o Claude Code vai criar

Ao executar essa instrução, o Claude Code vai criar dois elementos principais: o arquivo src/database.js com o código que inicializa o banco, e as tabelas dentro do arquivo database/agendamento.db.

O arquivo src/database.js contém a lógica que cria as tabelas caso elas não existam. Isso significa que, na primeira vez que o sistema rodar, as tabelas serão criadas automaticamente. Nas vezes seguintes, o arquivo já existe e as tabelas já estão lá.

Estrutura das tabelas
clientes: armazena nome, telefone e observações de cada cliente. O campo id é gerado automaticamente.
servicos: armazena o nome do serviço e sua duração em minutos. Exemplos: "Corte de cabelo, 30 min" ou "Consulta, 60 min".
agendamentos: conecta um cliente a um serviço em uma data e horário específicos. O campo status indica se o agendamento está ativo ou foi cancelado.

Como verificar se o banco foi criado corretamente

Após o Claude Code executar a instrução, verifique se o arquivo do banco foi criado. Peça ao agente:

Instrução para o Claude Code 
Verifique se o banco de dados foi criado corretamente.
Liste as tabelas existentes e mostre a estrutura de cada uma.

O agente vai executar um comando que mostra as tabelas e seus campos. Confira se as três tabelas aparecem (clientes, servicos, agendamentos) e se os campos correspondem ao que foi pedido.

Onde fica o arquivo e como fazer backup

O arquivo agendamento.db fica na pasta database/ dentro do projeto. Esse é o único arquivo que contém seus dados. Para fazer backup, basta copiar esse arquivo para outro local: um pendrive, uma pasta na nuvem ou outro diretório do computador.

O backup deve ser feito com o servidor parado. Feche o sistema antes de copiar o arquivo para garantir que nenhuma operação de escrita esteja em andamento.

Dica: faça um commit após configurar o banco

Com o banco configurado e funcionando, peça ao Claude Code para fazer um commit: "o banco de dados está configurado e as tabelas foram criadas, faça um commit". Isso cria um ponto de restauração caso algo quebre nas próximas etapas.

Criando o backend

O backend é a parte do sistema que você não vê. Ele recebe pedidos da interface (como "salve este cliente" ou "mostre os agendamentos de hoje"), processa esses pedidos e responde com os dados corretos.

O que é backend, em linguagem simples

Imagine um restaurante. O frontend é o salão: o que o cliente vê, os menus, as mesas. O backend é a cozinha: onde os pedidos são preparados. O cliente não vê a cozinha, mas sem ela não há comida.

No sistema de agendamento, o backend é o programa que roda no seu computador, recebe os dados que você digita na tela, salva no banco de dados e devolve as informações quando você pede para ver a agenda. Ele é construído com Node.js e Express, duas ferramentas amplamente usadas para esse tipo de trabalho.

Como pedir ao Claude Code para criar o servidor

A instrução para criar o backend deve ser detalhada. Quanto mais específica, melhor o resultado:

Instrução para o Claude Code 
Crie o servidor backend em src/server.js usando Express.
O servidor deve rodar na porta 3000 e ter as seguintes rotas:

Clientes:
- GET /api/clientes (listar todos)
- POST /api/clientes (criar novo)
- PUT /api/clientes/:id (editar)
- DELETE /api/clientes/:id (excluir)

Serviços:
- GET /api/servicos (listar todos)
- POST /api/servicos (criar novo)

Agendamentos:
- GET /api/agendamentos?data=YYYY-MM-DD (listar por dia)
- GET /api/agendamentos/semana?inicio=YYYY-MM-DD (listar semana)
- POST /api/agendamentos (criar novo)
- PUT /api/agendamentos/:id (editar)
- PATCH /api/agendamentos/:id/cancelar (cancelar)

O servidor deve servir arquivos estáticos da pasta public/.
Use o banco de dados configurado no capítulo anterior.

Entendendo as rotas

Cada rota é um endereço que a interface usa para se comunicar com o backend. O verbo antes da rota indica o tipo de operação:

Verbos HTTP
GET: buscar dados. "Me mostre a lista de clientes" ou "me mostre os agendamentos de hoje".
POST: criar algo novo. "Cadastre este cliente" ou "crie este agendamento".
PUT: atualizar algo existente. "Altere o telefone deste cliente" ou "mude o horário deste agendamento".
DELETE: excluir algo. "Remova este cliente do sistema".
PATCH: alterar parcialmente. "Mude apenas o status deste agendamento para cancelado".

Você não precisa memorizar isso. O importante é entender que cada ação na interface (clicar em "Salvar", abrir a agenda, cancelar um agendamento) corresponde a uma chamada para uma dessas rotas no backend.

Testando o backend com a skill testing-debugging

Antes de criar a interface, é importante verificar se o backend está funcionando. Acione a skill de testes e peça ao Claude Code para verificar:

Instrução para o Claude Code 
/skill testing-debugging

Teste todas as rotas do backend:
1. Crie um cliente de teste
2. Liste os clientes e verifique se o cliente aparece
3. Crie um serviço de teste
4. Crie um agendamento associando o cliente ao serviço
5. Liste os agendamentos do dia e verifique se aparece
6. Cancele o agendamento e verifique se o status mudou

O Claude Code vai executar cada teste e reportar se funcionou ou não. Se algum teste falhar, ele vai tentar corrigir o problema automaticamente. Se não conseguir, vai descrever o erro para que você possa fornecer mais contexto.

Lidando com erros nessa etapa

Os erros mais comuns ao criar o backend são: porta 3000 já em uso (outro programa está usando essa porta), pacotes não instalados (o Claude Code esqueceu de rodar npm install), ou erros de sintaxe no código gerado.

Se o servidor não iniciar, copie a mensagem de erro completa e envie ao Claude Code. Na maioria dos casos, a mensagem de erro contém informação suficiente para o agente identificar e corrigir o problema.

Dica: faça um commit após o backend funcionar

Quando todos os testes passarem, peça ao Claude Code para fazer um commit: "o backend está funcionando com todas as rotas testadas, faça um commit". Esse ponto de restauração é importante porque o próximo capítulo vai adicionar a interface, e se algo quebrar, você pode voltar ao backend funcional.

Criando o frontend

O frontend é a parte do sistema que você vê e interage. São as telas, os botões, os formulários. É onde você cadastra clientes, cria agendamentos e consulta a agenda.

O que é frontend, em linguagem simples

Se o backend é a cozinha do restaurante, o frontend é o salão. É o que o usuário vê: o cardápio, a mesa, o garçom anotando o pedido. No sistema de agendamento, o frontend são as páginas HTML que aparecem no navegador quando você acessa localhost:3000.

O frontend não guarda dados. Ele apenas mostra informações que busca no backend e envia dados que você digita para o backend salvar. A comunicação entre os dois acontece automaticamente, sem que você precise fazer nada além de clicar nos botões.

Usando a skill frontend-design

Antes de pedir a interface, acione a skill de frontend para garantir que o resultado tenha qualidade visual:

Dentro do Claude Code 
/skill frontend-design

Com a skill ativa, o Claude Code vai gerar interfaces com melhor organização visual, cores consistentes e layout responsivo. Sem ela, o resultado funciona, mas pode parecer desorganizado ou visualmente inconsistente.

As telas que o sistema precisa ter

O sistema de agendamento precisa de quatro telas principais:

Telas do sistema
Agenda do dia: a tela principal. Mostra todos os agendamentos do dia selecionado em ordem cronológica, com nome do cliente, serviço e horário.
Novo agendamento: formulário para criar um agendamento. Campos: cliente (selecionar da lista), serviço (selecionar da lista), data e horário.
Lista de clientes: mostra todos os clientes cadastrados com opções de editar, excluir e cadastrar novo.
Edição/cancelamento: ao clicar em um agendamento na agenda, poder editar seus dados ou cancelá-lo.

Crie um script para iniciar o sistema com um duplo clique.

Onde fica o arquivo do banco de dados e como fazer backup

O arquivo agendamento.db fica na raiz da pasta do projeto. Esse arquivo contém todos os seus dados. Para fazer backup, copie esse arquivo para outro local. Algumas opções práticas:

Opções de backup
Google Drive ou Dropbox: copie o arquivo para uma pasta sincronizada com a nuvem. O backup acontece automaticamente sempre que você copiar o arquivo.
Pendrive: copie o arquivo para um pendrive guardado em local seguro. Simples e não depende de internet.
HD externo: ideal para backups semanais. Conecte o HD, copie o arquivo, desconecte.
Outra pasta no mesmo computador: menos seguro (se o HD falhar, você perde tudo), mas melhor do que nenhum backup.

Faça backup pelo menos uma vez por semana. Se você usa o sistema intensamente, faça backup diário. O processo leva menos de um minuto.

Quando algo der errado

Erros fazem parte do desenvolvimento de software. Não são sinal de que algo está fundamentalmente errado: são parte normal do processo. O que importa é saber como lidar com eles.

Os tipos de erro mais comuns em projetos feitos com Claude Code

Projetos criados com Claude Code por pessoas sem experiência em programação tendem a ter três categorias de erro:

A primeira categoria são erros de ambiente: o Node.js não está instalado, a versão está desatualizada, um pacote não foi instalado corretamente. Esses erros geralmente aparecem logo no início, quando você tenta iniciar o servidor pela primeira vez. A mensagem de erro costuma mencionar o nome do pacote ou módulo que está faltando.

A segunda categoria são erros de lógica: o sistema faz algo diferente do que deveria. Um agendamento é salvo com a data errada, a lista de clientes não aparece, um botão não responde. Esses erros são mais sutis e às vezes só aparecem durante o uso real do sistema.

A terceira categoria são erros de integração: o frontend e o backend não estão se comunicando corretamente. A tela tenta buscar dados de uma rota que não existe, ou envia dados em um formato que o servidor não entende. Esses erros geralmente aparecem no console do navegador (acessível pressionando F12) ou no terminal onde o servidor está rodando.

Como descrever um erro ao Claude Code de forma que ele consiga corrigir

A qualidade da correção depende diretamente da qualidade da descrição do problema. Um erro bem descrito tem quatro elementos:

Os quatro elementos de um bom relato de erro
1. O que você fez: descreva a sequência de ações que levou ao erro. "Abri o sistema, fui para a tela de novo agendamento, preenchi todos os campos e cliquei em Salvar."
2. O que esperava: descreva o resultado esperado. "Esperava que o agendamento aparecesse na agenda do dia."
3. O que aconteceu: descreva o resultado real. "Em vez disso, a tela ficou em branco e não aconteceu nada."
4. A mensagem de erro: copie e cole qualquer mensagem de erro que aparecer, seja no terminal, seja na tela do navegador. Não resuma: copie o texto completo.

Quando o Claude Code não consegue corrigir sozinho

Em alguns casos, o Claude Code vai tentar corrigir um erro e a correção vai criar outro erro. Isso pode acontecer quando o problema é mais profundo do que parece na superfície, ou quando a instrução original foi ambígua e o agente tomou uma decisão que não era a ideal.

Se você perceber que o agente está tentando a mesma abordagem repetidamente sem sucesso, ou que cada correção cria um novo problema, pare. Não continue pedindo correções incrementais. Em vez disso, descreva o problema do início: o que o sistema deveria fazer, o que está fazendo de errado e o que já foi tentado para corrigir. Às vezes, recomeçar a descrição do problema do zero leva o agente a uma abordagem completamente diferente que resolve o problema de forma mais limpa.

Se o problema persistir após duas ou três tentativas com abordagens diferentes, considere desfazer as últimas mudanças e voltar ao estado anterior usando o Git, conforme explicado na próxima seção.

A importância de não acumular erros

Um erro não corrigido tende a gerar outros erros. Se você perceber que algo não está funcionando corretamente, corrija antes de continuar adicionando funcionalidades. Um sistema com cinco problemas acumulados é muito mais difícil de corrigir do que um sistema com um problema de cada vez.

A regra prática é: nunca avance para o próximo capítulo ou para a próxima funcionalidade com um erro conhecido não resolvido. Se algo não funciona, resolva agora. Se não conseguir resolver, peça ajuda ao Claude Code com uma descrição detalhada antes de continuar.

Como usar o Git para voltar a uma versão anterior

O Git é o sistema de controle de versão que você configurou no Capítulo 1. Ele registra o histórico de mudanças do projeto. Se algo quebrar de forma que você não consiga corrigir, você pode voltar ao estado anterior do projeto usando o Git.

Primeiro, veja o histórico de commits para identificar para qual versão você quer voltar:

Terminal 
git log --oneline

Esse comando mostra uma lista de commits com um código curto e uma descrição. Cada commit é um ponto no histórico do projeto. Identifique o commit que representa o estado em que o sistema estava funcionando corretamente.

Para voltar a esse estado sem perder o histórico, peça ao Claude Code:

Instrução para o Claude Code 
O sistema está com problemas que não consigo corrigir. Quero voltar ao estado
do commit [cole aqui o código do commit, ex: ab3f7c2]. Mostre o que vai mudar
antes de fazer qualquer alteração.

O agente vai mostrar o que vai ser revertido antes de fazer qualquer mudança. Confirme apenas quando tiver certeza de que é isso que você quer. Lembre-se: ao voltar para um commit anterior, as mudanças feitas depois daquele ponto serão perdidas. Por isso, faça commits frequentes durante o desenvolvimento, especialmente após cada funcionalidade que estiver funcionando corretamente.

Dica: faça commits frequentes

Sempre que uma funcionalidade estiver funcionando, peça ao Claude Code para fazer um commit: "o sistema de cadastro de clientes está funcionando corretamente, faça um commit com essa mensagem". Isso cria pontos de restauração ao longo do desenvolvimento.

O que você construiu

Você chegou ao final do tutorial com um sistema de agendamento funcionando no seu computador. Não um protótipo, não uma demonstração: um sistema real, que salva dados, que você pode abrir amanhã de manhã e usar para organizar sua agenda.

O sistema que você construiu faz o seguinte: cadastra clientes com nome, telefone e observações; cadastra serviços com nome e duração; cria agendamentos associando cliente, serviço, data e horário; mostra a agenda do dia em ordem cronológica; mostra a agenda da semana; permite editar e cancelar agendamentos; e guarda tudo em um arquivo local no seu computador que você pode copiar como backup a qualquer momento.

O que o sistema não faz

O sistema não envia notificações para os clientes. Não tem controle de acesso ou login. Não gera relatórios financeiros. Não funciona em outros computadores sem ser instalado neles. Não está disponível na internet. Essas limitações são intencionais: elas mantêm o sistema simples, confiável e fácil de manter.

Para o uso que foi proposto, um sistema local de agendamentos para o dono do negócio, essas funcionalidades são suficientes. Um sistema que faz poucas coisas e as faz bem é mais útil do que um sistema que tenta fazer tudo e funciona mal.
