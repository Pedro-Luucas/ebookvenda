Introdução

O que mudou e o que você vai construir

Durante décadas, criar um produto de software exigiu uma combinação específica de habilidades: saber programar, entender arquitetura de sistemas, dominar bancos de dados, configurar servidores. Quem não tinha esse conjunto de conhecimentos ficava dependente de contratar desenvolvedores, o que tornava o custo de entrada proibitivo para a maioria das pessoas com boas ideias.

Isso mudou. Não de forma gradual, mas de forma abrupta, nos últimos dois anos. Ferramentas como o Claude Code tornaram possível que uma pessoa sem nenhum conhecimento de programação descreva o que quer construir e receba, em minutos, código funcional, estruturado e pronto para ser publicado na internet. O que antes levava semanas de trabalho de um desenvolvedor experiente agora pode ser feito em horas por alguém que nunca escreveu uma linha de código.

Este ebook parte desse ponto. Você já sabe usar o Claude Code. O que este guia vai fazer é mostrar como usar essa ferramenta para construir algo que vai além de um experimento local: um SaaS real, publicado na internet, com autenticação de usuários, banco de dados e uma landing page para apresentar o produto.

O que você vai construir

Ao final deste ebook, você terá um SaaS funcional publicado na internet. Não um protótipo rodando no seu computador. Não um projeto de demonstração. Um produto real, acessível por qualquer pessoa com um navegador, com usuários podendo criar conta, fazer login e usar o que você construiu.

A stack técnica usada é Next.js com App Router para o frontend e backend, Supabase para banco de dados e autenticação, e Vercel para o deploy. O Claude Code vai configurar tudo isso. Você não precisa entender cada tecnologia em profundidade para seguir este guia.

O que está fora do escopo

Este ebook não cobre pagamentos, marketing ou crescimento pós-lançamento. Esses são temas importantes, mas cada um deles merece um guia próprio. O foco aqui é chegar ao produto publicado e funcionando. Pagamentos podem ser integrados depois, quando você já tiver usuários reais usando o produto.

O pré-requisito para este guia é saber o básico do Claude Code: como abrir um projeto, como dar instruções, como revisar o que foi feito. Se você ainda não tem esse conhecimento, leia o guia introdutório do Claude Code antes de continuar.

O erro que mata a maioria dos projetos antes de começar

A maioria das pessoas que decide criar um SaaS começa pelo lugar errado. Não por falta de capacidade, mas por um impulso natural: quando você tem uma ferramenta nova e poderosa nas mãos, a vontade é usá-la imediatamente.

O impulso de ir direto para o código

Quando alguém descobre que pode descrever um produto ao Claude Code e receber código funcional em minutos, a reação mais comum é começar a construir na mesma hora. A ideia aparece, a empolgação chega junto, e a próxima ação parece óbvia: abrir o Claude Code e começar a pedir telas, funcionalidades, banco de dados.

O problema não é a velocidade. O problema é que construir rápido sem saber o que construir não economiza tempo. Multiplica o retrabalho. Você passa dias pedindo ao Claude Code para criar funcionalidades que depois percebe que não fazem sentido para o problema que quer resolver. Pede para refazer. Muda de direção. Pede de novo. O projeto cresce em complexidade sem crescer em clareza.

Esse padrão é tão comum que tem nome na comunidade de desenvolvedores independentes: construir em busca de um problema. Você cria a solução antes de entender o problema com precisão suficiente para saber se a solução resolve alguma coisa.

O que acontece quando você constrói sem definir o problema

Imagine que você decide criar "um app de produtividade". Você pede ao Claude Code para criar um sistema de tarefas com categorias, prioridades, lembretes e integração com calendário. Duas semanas depois, você tem um produto funcionando. Mas quando tenta explicar para alguém o que ele faz de diferente dos outros apps de produtividade que já existem, não consegue responder com clareza.

O produto existe, mas não tem um problema específico que resolve melhor do que qualquer alternativa. Não tem um grupo específico de pessoas para quem foi feito. Não tem uma razão clara para alguém pagar por ele em vez de usar algo gratuito que já existe.

Isso não é um problema de execução. É um problema de definição. E ele não aparece no código. Aparece quando você tenta vender.

A diferença entre uma ideia e um produto

Uma ideia é uma direção. "Quero criar algo para ajudar freelancers a gerenciar clientes" é uma ideia. Um produto é uma solução específica para um problema específico de um grupo específico de pessoas. "Um sistema simples para freelancers de design gráfico registrarem horas por projeto e enviarem propostas de orçamento com um clique" é um produto.

A diferença entre os dois não é o nível de detalhe técnico. É a clareza sobre quem tem o problema, qual é o problema exato e por que a solução que você vai construir resolve melhor do que o que já existe.

Sem essa clareza, você pode construir qualquer coisa. Com ela, você sabe exatamente o que construir, o que deixar de fora e como explicar o produto para alguém em trinta segundos.

Por que "eu quero criar um app" não é suficiente

A frase "eu quero criar um app" não contém nenhuma informação útil para começar a construir. Ela diz o formato, mas não diz o problema, o público, o contexto de uso, o que o produto faz de diferente ou por que alguém pagaria por ele.

Antes de abrir o Claude Code para escrever a primeira linha de código, você precisa ter respostas claras para quatro perguntas: Quem tem o problema? Qual é o problema exato? Por que esse problema não está sendo resolvido bem pelas soluções que já existem? O que a sua solução vai fazer de diferente?

Essas perguntas não são formalidade. São o filtro que separa projetos que chegam ao ar de projetos que ficam eternamente em construção.

O que precisa estar definido antes de escrever código

Antes de abrir o Claude Code para construir qualquer coisa, você precisa ter três coisas definidas: o nicho (quem tem o problema e qual é o problema), o produto (o que você vai construir para resolver esse problema) e o escopo da versão inicial (o que entra na primeira versão e o que fica para depois).

Os próximos três capítulos cobrem exatamente isso, nessa ordem. Não pule para o capítulo 6 antes de ter essas três coisas definidas. O tempo que você gasta planejando antes de construir é o tempo que você economiza não precisando refazer.

Atenção

O Claude Code é muito bom em construir o que você pede. Mas ele não sabe o que você deveria pedir. Essa parte é sua. E é a parte mais importante de todo o processo.

Encontrando seu nicho

Este é o capítulo mais importante do ebook. Não porque seja o mais técnico, mas porque é onde a maioria das pessoas trava. Encontrar um nicho não é uma questão de criatividade. É uma questão de método.

O que é um nicho, em linguagem simples

Um nicho é a combinação de um problema específico com um grupo específico de pessoas que tem esse problema. Não é um mercado amplo como "produtividade" ou "saúde". É algo como "nutricionistas que atendem por videochamada e precisam enviar planos alimentares personalizados para cada paciente sem gastar horas formatando documentos".

Quanto mais específico o nicho, mais fácil é construir algo que realmente resolve o problema. E mais fácil é encontrar as pessoas certas para usar o produto. Um nicho pequeno e bem definido é mais valioso do que um nicho grande e vago, especialmente para quem está construindo sozinho.

A lógica é simples: se você tenta resolver um problema para todo mundo, acaba não resolvendo bem para ninguém. Se você resolve um problema específico para um grupo específico de pessoas, você tem uma chance real de ser a melhor opção disponível para esse grupo.

Por que nicho pequeno é melhor para quem está começando

Existe uma tendência natural de querer construir algo grande, que sirva para muitas pessoas. A lógica parece correta: quanto maior o mercado, maior o potencial de receita. Mas essa lógica ignora o custo de competir em mercados grandes.

Em mercados grandes, você compete com empresas que têm equipes de dezenas de pessoas, orçamentos de marketing expressivos e anos de vantagem. Em nichos pequenos, você compete com planilhas, processos manuais e soluções genéricas que não foram feitas para aquele grupo específico. Essa é uma competição muito mais favorável.

Além disso, nichos pequenos têm uma característica valiosa: as pessoas que fazem parte deles se conhecem. Um nutricionista satisfeito com o seu produto vai recomendar para outros nutricionistas. Esse tipo de crescimento orgânico é muito mais eficiente do que qualquer campanha de marketing.

Os três critérios de um bom nicho

Antes de decidir por um nicho, ele precisa passar por três critérios. Se falhar em qualquer um dos três, o nicho não é viável para um SaaS.

Primeiro: o problema existe de verdade. Não é uma suposição sua sobre o que as pessoas precisam. É algo que você pode verificar: as pessoas falam sobre esse problema em fóruns, grupos, redes sociais. Elas buscam soluções. Elas reclamam das soluções que existem.

Segundo: as pessoas pagam para resolver. Existem problemas reais que as pessoas preferem conviver a pagar para resolver. Um SaaS precisa de um problema que as pessoas considerem urgente o suficiente para justificar um pagamento mensal. A forma mais simples de verificar isso é ver se já existem produtos pagos nesse espaço. Se existem, as pessoas pagam. Se não existem, pode ser que o problema não seja urgente o suficiente.

Terceiro: você consegue construir a solução. Com o Claude Code, o escopo do que é possível construir expandiu muito. Mas ainda existem limites. Integrações complexas com sistemas legados, processamento de dados em tempo real em escala, algoritmos de machine learning sofisticados: esses são exemplos de coisas que estão fora do escopo de um primeiro SaaS construído com Claude Code. Escolha um problema cuja solução seja um aplicativo web com banco de dados, autenticação e lógica de negócio razoavelmente simples.

Como avaliar um nicho sem sair do computador

Você não precisa fazer entrevistas com dezenas de pessoas para validar um nicho. Existem formas de fazer uma avaliação inicial usando apenas o que está disponível online.

Procure por grupos no Facebook, comunidades no Reddit, fóruns especializados e canais no YouTube voltados para o grupo de pessoas que você quer atender. Observe o que elas reclamam. Veja quais perguntas aparecem com frequência. Preste atenção nas soluções que elas usam hoje e no que dizem sobre essas soluções.

Procure por produtos que já existem nesse espaço. Leia as avaliações negativas. As avaliações negativas de produtos concorrentes são um mapa do que as pessoas precisam e não estão recebendo. Se você encontrar um padrão de reclamações sobre algo específico, esse é um sinal de oportunidade.

Procure por pessoas que cobram para fazer manualmente o que você quer automatizar. Se existem freelancers ou consultores que cobram para fazer algo que poderia ser feito por um software, isso é um sinal forte de que o problema existe e que as pessoas pagam para resolver.

Exemplos de nichos que funcionam bem para SaaS pequeno

Alguns padrões de nicho têm funcionado bem para produtos pequenos construídos por uma pessoa só. Não são receitas, mas referências para calibrar o tipo de problema que vale a pena explorar.

Profissionais autônomos que precisam gerenciar clientes, projetos ou documentos: advogados, nutricionistas, psicólogos, designers, consultores. Esses grupos têm processos repetitivos que consomem tempo e que as ferramentas genéricas não resolvem bem.

Pequenos negócios com processos específicos do setor: pet shops que precisam gerenciar agendamentos de banho e tosa, academias que precisam controlar frequência de alunos, clínicas que precisam enviar lembretes de consulta. Cada setor tem suas particularidades, e ferramentas genéricas raramente as atendem bem.

Criadores de conteúdo com necessidades operacionais: pessoas que gerenciam múltiplos canais, que precisam organizar pautas, que trabalham com patrocinadores e precisam controlar entregas e pagamentos.

Como usar o Claude Code para explorar e comparar nichos

O Claude Code pode ser um parceiro valioso na fase de exploração de nichos, antes de qualquer linha de código. Você pode usá-lo para analisar ideias, identificar pontos fracos em um nicho e comparar opções. A chave é formular as perguntas de forma específica.

Dica prática

Se você já faz parte de um grupo profissional ou tem experiência em algum setor, comece por aí. Você já conhece os problemas, já sabe o que as pessoas reclamam e já tem credibilidade para conversar com potenciais usuários. Construir para um grupo do qual você faz parte é uma vantagem real.

Definindo o produto

Você tem um nicho. Agora precisa transformar esse nicho em um produto. Essa transição parece simples, mas é onde muitos projetos perdem o foco. Definir um produto é, em grande parte, um exercício de corte: decidir o que não vai entrar.

Da ideia ao escopo

Quando você tem um problema claro e um grupo de pessoas bem definido, a tentação é listar tudo que o produto poderia fazer para resolver esse problema. Um sistema para nutricionistas poderia ter cadastro de pacientes, histórico de consultas, geração de planos alimentares, cálculo de macronutrientes, integração com aplicativos de saúde, lembretes automáticos, relatórios de progresso, chat com o paciente, assinatura digital de documentos.

Cada uma dessas funcionalidades faz sentido individualmente. O problema é que construir todas elas na primeira versão vai levar meses, e você vai lançar um produto complexo para um mercado que ainda não conhece você. Enquanto isso, o problema que você quer resolver continua sem solução para as pessoas que precisam dela.

Definir o produto significa escolher o subconjunto mínimo de funcionalidades que resolve o problema central. Não o problema completo. O problema central.

O conceito de MVP

MVP é a sigla em inglês para "produto mínimo viável" (minimum viable product). O conceito é simples: a versão mais enxuta do produto que ainda entrega valor real para o usuário. Não é um protótipo. Não é uma demonstração. É um produto que funciona, que resolve um problema real, mas que tem apenas as funcionalidades essenciais para fazer isso.

A palavra "viável" é importante. Viável não significa perfeito. Significa que o produto funciona bem o suficiente para que alguém use e pague por ele. Um MVP ruim é aquele que tem tantas funcionalidades que demora meses para ficar pronto. Um MVP bom é aquele que você consegue construir em semanas e que já resolve o problema central do usuário.

Para o sistema de nutricionistas, o MVP poderia ser apenas: cadastro de pacientes, registro de consultas e geração de um plano alimentar em PDF. Tudo o mais fica para versões futuras, depois que você tiver usuários reais usando o produto e dando feedback.

O que entra na versão 1 e o que fica para depois

Uma forma prática de decidir o que entra no MVP é fazer duas listas. Na primeira, coloque tudo que o produto poderia fazer. Na segunda, coloque apenas o que o produto precisa fazer para que alguém use e pague por ele hoje.

A segunda lista é o seu MVP. Tudo que está na primeira lista mas não na segunda é uma funcionalidade futura. Não descarte essas ideias, mas não as construa agora. Guarde-as para depois que você tiver validado que o produto resolve o problema central.

Uma regra útil: se uma funcionalidade não impede o usuário de resolver o problema central sem ela, ela não entra no MVP. O usuário pode resolver o problema central com um processo manual enquanto você não constrói essa funcionalidade? Se sim, ela fica para depois.

Como escrever a definição do produto antes de qualquer código

Antes de pedir ao Claude Code para construir qualquer coisa, escreva a definição do produto em linguagem simples. Não em linguagem técnica. Em linguagem que qualquer pessoa entenderia.

A definição deve responder: para quem é o produto, qual problema ele resolve, o que o usuário consegue fazer com ele e o que não está incluído na versão inicial. Esse documento não precisa ser longo. Dois ou três parágrafos são suficientes. O que importa é a clareza.

Instrução para o Claude Code: definir o escopo do produto 
Quero criar um SaaS para [grupo de pessoas] que resolve [problema específico].

Com base nessa descrição, me ajude a definir o escopo do produto:

1. Quais são as 3 a 5 funcionalidades essenciais para a versão 1?
2. Quais funcionalidades fazem sentido mas devem ficar para versões futuras?
3. Como seria o fluxo principal do usuário: o que ele faz desde que entra
   no produto até resolver o problema?
4. Quais telas o produto precisa ter na versão 1?

Seja conservador. Prefira um escopo menor que funcione bem a um escopo
maior que demore meses para ficar pronto.

O que a definição do produto deve conter
Para quem: o grupo específico de pessoas que vai usar o produto.
O problema: o que essas pessoas precisam resolver e não conseguem resolver bem hoje.

O que o produto não faz: o que fica explicitamente fora do escopo inicial.

Planejando antes de construir

Você tem o nicho, tem o produto definido, tem o escopo da versão 1. Agora vem a etapa que separa projetos que ficam coerentes do início ao fim de projetos que se perdem no meio do caminho: o CLAUDE.md.

O que é o CLAUDE.md e por que ele importa

O CLAUDE.md é um arquivo de texto que fica na raiz do seu projeto e serve como documento de referência para o Claude Code. Toda vez que você abre o Claude Code em um projeto, ele lê esse arquivo antes de qualquer coisa. É nele que você descreve o que o projeto é, como ele está estruturado, quais tecnologias usa, quais são as convenções e o que o produto deve fazer.

Sem um CLAUDE.md bem escrito, o Claude Code trabalha sem contexto. Ele vai fazer escolhas técnicas baseadas em padrões genéricos, pode usar tecnologias diferentes das que você quer, pode criar estruturas que não fazem sentido para o seu produto. Com um CLAUDE.md bem escrito, ele tem um mapa claro do que você está construindo e como.

Pense no CLAUDE.md como a conversa que você teria com um desenvolvedor no primeiro dia de trabalho. Você explicaria o produto, a stack técnica, as convenções do projeto e o que está sendo construído. O CLAUDE.md faz isso de forma permanente, para que você não precise repetir essas informações toda vez que abre uma nova sessão.

O que incluir em um CLAUDE.md eficaz

Um bom CLAUDE.md para um SaaS tem cinco seções principais. Cada uma serve a um propósito específico e influencia diretamente a qualidade do código que o Claude Code vai produzir.

Descrição do projeto: o que o produto é, para quem é e qual problema resolve. Duas ou três frases são suficientes. O objetivo é dar contexto imediato sobre o propósito do que está sendo construído.

Stack técnica: quais tecnologias o projeto usa e como elas se relacionam. Para o nosso caso: Next.js com App Router, TypeScript, Tailwind CSS, Supabase para banco de dados e autenticação, Vercel para deploy. Especifique versões quando relevante.

Estrutura do projeto: como as pastas e arquivos estão organizados. Quais são as rotas principais, onde ficam os componentes, onde ficam as funções de acesso ao banco de dados.

Funcionalidades e telas: o que o produto faz, quais são as telas principais e o que cada uma contém. Quanto mais específico, melhor. Não escreva "tela de dashboard". Escreva "tela de dashboard que mostra os últimos 10 pacientes cadastrados, com nome, data da última consulta e um botão para acessar o perfil".

Convenções: regras que o Claude Code deve seguir ao escrever código. Idioma dos textos da interface, padrões de nomenclatura, como tratar erros, como estruturar as rotas de API.

Exemplo completo de um CLAUDE.md para um SaaS

O exemplo abaixo é para um SaaS fictício de gestão de pacientes para nutricionistas. Use-o como referência para estruturar o seu próprio CLAUDE.md, adaptando cada seção para o seu produto.

Exemplo de CLAUDE.md 
# NutriGestão

## Sobre o Projeto
SaaS para nutricionistas autônomos gerenciarem pacientes e consultas.
O nutricionista cadastra pacientes, registra consultas e gera planos
alimentares em PDF. Toda a interface está em português brasileiro.

## Stack Técnica
- Frontend e backend: Next.js 15 com App Router, TypeScript, Tailwind CSS
- Banco de dados e autenticação: Supabase
- Deploy: Vercel
- Path alias: @/* aponta para a raiz do projeto

## Estrutura
- app/page.tsx: landing page pública
- app/(auth)/: rotas de login e cadastro
- app/(dashboard)/: área autenticada do nutricionista
- app/api/: rotas de API server-side
- app/components/: componentes reutilizáveis
- app/lib/supabase/: cliente Supabase e funções de acesso ao banco

## Funcionalidades da Versão 1
1. Autenticação: cadastro e login via Supabase Auth (email e senha)
2. Pacientes: listar, cadastrar, editar e arquivar pacientes
3. Consultas: registrar consultas por paciente com data e anotações
4. Plano alimentar: criar e salvar plano alimentar por paciente

## Telas Principais
- /login: formulário de login
- /cadastro: formulário de cadastro do nutricionista
- /dashboard: lista de pacientes com busca e filtro
- /pacientes/novo: formulário de cadastro de paciente
- /pacientes/[id]: perfil do paciente com histórico de consultas
- /pacientes/[id]/consulta/nova: formulário de nova consulta
- /pacientes/[id]/plano: editor de plano alimentar

## Convenções
- Toda interface visível ao usuário em português brasileiro
- Componentes que usam hooks do browser têm "use client" no topo
- Server Components são o padrão para páginas e layouts
- Erros de formulário exibidos inline, abaixo do campo com problema
- Rotas de API retornam JSON com { data } em caso de sucesso
  e { error } em caso de falha

Como descrever funcionalidades com precisão

A parte mais importante do CLAUDE.md é a descrição das funcionalidades e telas. Quanto mais específica for essa descrição, menos retrabalho você vai ter. O Claude Code vai construir exatamente o que você descreveu, então vale a pena gastar tempo escrevendo com precisão.

Evite descrições vagas como "tela de perfil do usuário". Prefira descrições que incluam o que a tela mostra, quais ações o usuário pode fazer e quais dados são exibidos. Se uma tela tem um formulário, descreva os campos. Se uma tela tem uma lista, descreva o que aparece em cada item da lista.

Instrução para o Claude Code: criar o CLAUDE.md

Skills que você vai precisar

O Claude Code tem capacidades padrão que cobrem a maioria das tarefas de desenvolvimento. Mas para um projeto SaaS com qualidade de produção, algumas skills específicas fazem diferença real na qualidade do resultado.

Por que skills importam neste projeto

Skills são extensões do Claude Code que adicionam conhecimento especializado e comportamentos específicos para determinados tipos de tarefa. Quando você ativa uma skill, o Claude Code passa a seguir um conjunto de práticas e padrões otimizados para aquela área.

Para um SaaS construído com Next.js, Supabase e Vercel, existem cinco skills que vão fazer diferença direta na qualidade do que é construído. Cada uma resolve um problema específico que aparece com frequência nesse tipo de projeto.

As cinco skills recomendadas

react-nextjs-development

Esta skill garante que o Claude Code use os padrões corretos do Next.js com App Router. Sem ela, o Claude Code pode misturar padrões antigos do Pages Router com o App Router, criar componentes client-side onde deveriam ser server-side, ou estruturar as rotas de forma incorreta. Com ela ativa, o projeto vai seguir as convenções do Next.js 15 desde o início, o que evita uma categoria inteira de problemas difíceis de corrigir depois.

frontend-design

Esta skill melhora a qualidade visual das interfaces geradas. Sem ela, o Claude Code tende a criar interfaces funcionais mas com aparência genérica. Com ela ativa, o resultado tem mais cuidado com tipografia, espaçamento, hierarquia visual e consistência. Para um SaaS que vai ser apresentado a usuários reais, a aparência importa.

testing-debugging

Esta skill é essencial para identificar e corrigir erros sem precisar entender o código. Quando algo quebra, você descreve o problema e a skill orienta o Claude Code a diagnosticar a causa raiz de forma sistemática, em vez de tentar correções aleatórias. Para quem não sabe programar, essa skill é especialmente valiosa porque transforma erros incompreensíveis em problemas resolvíveis.

simplify

Esta skill revisa o código gerado em busca de complexidade desnecessária e o simplifica. Projetos construídos com IA tendem a acumular código redundante ao longo do tempo, especialmente quando você pede muitas alterações incrementais. A skill simplify mantém o código limpo e fácil de manter, o que reduz a chance de erros futuros.

typescript-expert

Esta skill garante que o TypeScript seja usado corretamente no projeto. TypeScript é a linguagem usada no Next.js e oferece verificação de tipos que previne uma classe inteira de erros. Sem essa skill, o Claude Code pode usar TypeScript de forma superficial, perdendo os benefícios da verificação de tipos. Com ela, o código vai ser mais robusto e os erros vão aparecer antes de chegar ao usuário.

Construindo o SaaS

Com o projeto configurado, o Supabase conectado e o CLAUDE.md no lugar, você está pronto para construir. O processo de construção com Claude Code segue um ritmo específico: descrever, revisar, aprovar, iterar.

O fluxo de construção com Claude Code

Cada funcionalidade do seu SaaS vai ser construída em ciclos. Você descreve o que quer, o Claude Code constrói, você revisa o resultado no navegador, aprova ou pede ajustes, e passa para a próxima funcionalidade. Esse ciclo se repete até o produto estar completo.

A regra mais importante desse fluxo é construir uma funcionalidade de cada vez. Não peça ao Claude Code para criar o sistema de autenticação, o dashboard e o cadastro de pacientes em uma única instrução. Peça um de cada vez, revise cada um antes de continuar. Isso mantém o projeto coerente e facilita a identificação de problemas.

A ordem recomendada para construir um SaaS é: autenticação primeiro, depois as telas principais, depois as rotas de API, depois as integrações com o banco de dados. Essa ordem garante que a base do produto esteja sólida antes de construir as funcionalidades específicas.

Publicando na internet com Vercel

Publicar o produto na internet é o momento em que ele deixa de ser um projeto local e se torna algo real. O processo com Vercel é direto, mas tem alguns detalhes que fazem diferença.

O que acontece no deploy

Deploy é o processo de pegar o código que está no seu computador (ou no GitHub) e publicá-lo em servidores que ficam acessíveis pela internet. Quando você faz um deploy no Vercel, ele pega o código do seu repositório no GitHub, executa o processo de build do Next.js (que transforma o código TypeScript em arquivos otimizados para produção) e publica o resultado em servidores distribuídos globalmente.

O resultado é uma URL pública, no formato nome-do-projeto.vercel.app, que qualquer pessoa com acesso à internet pode acessar. Toda vez que você enviar uma nova versão do código para o GitHub, o Vercel vai automaticamente fazer um novo deploy com as alterações.

Como fazer o primeiro deploy

Se você seguiu o capítulo anterior e conectou o projeto ao Vercel, o primeiro deploy já deve ter acontecido automaticamente quando você enviou o código para o GitHub. Para verificar, acesse o painel do Vercel e veja se o projeto aparece com status "Ready".

Se o deploy ainda não aconteceu, ou se você quer forçar um novo deploy, peça ao Claude Code para enviar as alterações para o GitHub:

Instrução para o Claude Code: fazer deploy 
Faça commit de todas as alterações pendentes e envie para o GitHub.
Use uma mensagem de commit descritiva sobre o que foi alterado.
Após o push, o Vercel vai fazer o deploy automaticamente.

Configurando variáveis de ambiente no Vercel

As variáveis de ambiente que você tem no arquivo .env.local no seu computador não são enviadas para o GitHub por segurança. Você precisa adicioná-las manualmente no painel do Vercel.

No painel do Vercel, acesse o projeto, clique em "Settings" e depois em "Environment Variables". Adicione cada variável com o nome e valor exatos. Para o Supabase, você precisa de NEXT_PUBLIC_SUPABASE_URL e NEXT_PUBLIC_SUPABASE_ANON_KEY. Se o seu produto tiver outras variáveis de ambiente, adicione todas aqui.

Após adicionar as variáveis, faça um novo deploy para que elas entrem em vigor. No painel do Vercel, clique em "Deployments", selecione o último deploy e clique em "Redeploy".

Como acessar o SaaS pelo domínio gerado

Após o deploy bem-sucedido, o Vercel vai mostrar a URL do seu produto. Acesse essa URL no navegador e teste o produto como se fosse um usuário real. Crie uma conta, faça login, use as funcionalidades principais.

Como configurar um domínio próprio

Se você tem um domínio próprio (como meuproduto.com.br), pode configurá-lo no Vercel para que o produto seja acessível por esse endereço. No painel do Vercel, acesse "Settings" e depois "Domains". Adicione o domínio e siga as instruções para configurar os registros DNS no seu provedor de domínio.

O que fazer quando o deploy falha

Quando um deploy falha, o Vercel mostra os logs do erro. Esses logs são a informação mais importante para diagnosticar o problema. Copie o erro e descreva ao Claude Code:

Instrução para o Claude Code: corrigir erro de deploy 
O deploy no Vercel falhou com o seguinte erro:

[cole aqui o erro completo dos logs do Vercel]

Analise o erro e corrija o problema. Após a correção, faça commit
e push para o GitHub para acionar um novo deploy.
Atenção

Os erros de build no Vercel são diferentes dos erros que aparecem durante o desenvolvimento local. O ambiente de produção é mais restrito. Erros de TypeScript que são ignorados localmente vão impedir o build no Vercel. Se o produto funciona localmente mas o deploy falha, o erro está quase sempre nos logs do Vercel.

Criando a landing page

O produto está publicado. Agora você precisa de uma forma de apresentá-lo para quem ainda não o conhece. A landing page é essa apresentação. Ela existe antes de qualquer divulgação e precisa responder uma pergunta em segundos: o que é isso e por que eu deveria usar?

Por que a landing page importa antes de qualquer divulgação

Quando você começa a mostrar o produto para pessoas, seja em grupos, redes sociais ou conversas diretas, a primeira coisa que elas vão fazer é acessar o endereço do produto. O que elas encontram nesse momento determina se vão continuar explorando ou fechar a aba.

Uma landing page bem feita não precisa ser longa nem sofisticada. Ela precisa ser clara. Quem é o produto, para quem é, o que ele faz e como começar a usar. Essas quatro informações, apresentadas de forma limpa e direta, são suficientes para uma primeira versão.

A landing page fica na rota raiz do projeto (/), que é a página pública acessível por qualquer pessoa antes de fazer login. Ela vai ser construída dentro do mesmo projeto Next.js, o que significa que vai ser publicada automaticamente no mesmo domínio do produto.

O que uma boa landing page de SaaS precisa ter

Para um produto que está começando, a landing page precisa de cinco seções. Cada uma tem um propósito específico e deve ser escrita pensando no visitante que nunca ouviu falar do produto.

Hero: a primeira coisa que o visitante vê. Deve ter o nome do produto, uma frase que explica o que ele faz (em uma linha), e um botão de chamada para ação (geralmente "Criar conta grátis" ou "Começar agora"). Sem imagens desnecessárias, sem texto longo.

Problema: uma descrição breve do problema que o produto resolve. Essa seção cria identificação com o visitante que tem esse problema. Duas ou três frases são suficientes.

Solução: como o produto resolve o problema. Pode ser uma lista de três a cinco funcionalidades principais, cada uma com uma frase de descrição. Não liste todas as funcionalidades. Liste as que mais importam para o usuário.

Como funciona: um fluxo simples de três passos mostrando como o usuário vai do problema à solução usando o produto. "Crie sua conta, cadastre seus pacientes, gere planos alimentares em segundos."

Chamada para ação final: um botão de cadastro no final da página, para o visitante que chegou até aqui e quer começar.

Como pedir ao Claude Code para criar a landing page

A landing page deve ser criada em app/page.tsx, que já existe no projeto. Peça ao Claude Code para substituir o conteúdo padrão pela landing page do seu produto:

Instrução para o Claude Code: criar a landing page 
Crie a landing page do produto em app/page.tsx.

Informações do produto:
- Nome: [nome do produto]
- Para quem é: [grupo de pessoas]
- O que resolve: [problema em uma frase]
- Funcionalidades principais: [liste 3 a 5]
- Como funciona: [descreva em 3 passos]

A landing page deve ter as seguintes seções:
1. Hero: nome, frase de valor e botão "Criar conta grátis" que leva para /cadastro
2. Problema: descrição do problema que o produto resolve
3. Solução: lista das funcionalidades principais com ícones
4. Como funciona: 3 passos visuais
5. CTA final: botão de cadastro

Use a skill frontend-design para garantir uma aparência profissional.
Toda a interface em português brasileiro.
O design deve ser limpo, com fundo branco e a cor de destaque do projeto.

Como revisar e ajustar o visual

Após criar a landing page, abra o produto no navegador e avalie o resultado. Verifique se o texto está claro, se a hierarquia visual está correta (o que é mais importante aparece maior e mais destacado), se os botões estão visíveis e se a página funciona bem em telas menores.

Para ajustes visuais, descreva o que você quer mudar de forma específica. "O botão do hero está pequeno demais, aumente o tamanho e use a cor azul do projeto" é uma instrução que o Claude Code consegue executar com precisão.

O que não colocar na landing page de um produto que está começando

Evite depoimentos de clientes que você ainda não tem. Evite números de usuários ou métricas que não existem. Evite promessas que o produto ainda não cumpre. Evite seções de preços se você ainda não definiu o modelo de cobrança.

Uma landing page honesta e direta é mais eficaz do que uma landing page cheia de elementos que não têm substância. O visitante percebe quando algo não é genuíno, e isso prejudica a confiança no produto antes mesmo de ele ser testado.

Dica de revisão

Mostre a landing page para alguém que não conhece o produto e peça para essa pessoa explicar o que o produto faz depois de ler a página. Se a explicação estiver correta, a landing page está clara. Se estiver errada ou confusa, o texto precisa ser revisado.

Quando algo der errado

Algo vai dar errado. Não é pessimismo, é a realidade de qualquer projeto de software. A diferença entre quem avança e quem trava está em saber como responder quando isso acontece.

Os erros mais comuns em projetos SaaS com Claude Code

Alguns erros aparecem com tanta frequência que vale a pena conhecê-los antes de encontrá-los. Cada um tem uma causa típica e uma forma de resolver.

O produto funciona localmente mas não no Vercel. Quase sempre é um problema de variáveis de ambiente não configuradas no painel do Vercel, ou um erro de TypeScript que o ambiente local ignora mas o build de produção não. Verifique os logs do Vercel primeiro.

Os dados não aparecem na tela. Pode ser um problema de RLS (Row Level Security) no Supabase, onde as políticas de segurança estão bloqueando o acesso. Pode também ser um problema na query, onde os dados estão sendo buscados de forma incorreta. Verifique o painel do Supabase para ver se os dados existem no banco.

O login não funciona. Verifique se o Supabase Auth está configurado corretamente, se as URLs de redirecionamento estão configuradas no painel do Supabase (em Authentication, depois URL Configuration) e se o middleware está funcionando.

A tela fica em branco ou mostra um erro 500. Abra o console do navegador (F12) e veja o erro. Copie o erro completo e descreva ao Claude Code.

O Claude Code fez uma alteração que quebrou algo que funcionava. Use o Git para voltar à versão anterior (veja a seção abaixo) e descreva o problema com mais precisão antes de tentar novamente.

Como descrever um erro ao Claude Code de forma eficaz

A forma como você descreve um erro determina a qualidade da solução. Um erro bem descrito tem quatro elementos: onde o erro acontece (qual tela, qual ação), o que você esperava que acontecesse, o que aconteceu de fato, e o erro exato (do console do navegador ou dos logs do servidor).

Modelo para descrever um erro Encontrei um erro e preciso de ajuda para resolver. Onde: [nome da tela ou rota, ex: /pacientes/novo] Ação: [o que eu fiz, ex: cliquei no botão "Salvar" com o formulário preenchido] Esperado: [o que deveria acontecer, ex: o paciente deveria ser salvo e eu deveria ser redirecionado para /dashboard] O que aconteceu: [o que aconteceu de fato, ex: a página ficou em branco] Erro no console: [cole o erro exato aqui] Corrija o problema sem alterar outras partes do código que estão funcionando.

A importância de corrigir um erro de cada vez

Quando vários erros aparecem ao mesmo tempo, a tentação é pedir ao Claude Code para corrigir todos de uma vez. Resista a essa tentação. Corrija um erro de cada vez, verifique se a correção funcionou, e só então passe para o próximo.

Corrigir múltiplos erros de uma vez aumenta a chance de uma correção interferir na outra, criando novos problemas. O processo é mais lento no curto prazo, mas muito mais confiável.

Como usar o Git para voltar a uma versão anterior

O Git mantém um histórico de todas as versões do projeto. Se uma alteração quebrou algo que funcionava, você pode voltar à versão anterior. Peça ao Claude Code para fazer isso:

Instrução para o Claude Code: voltar a uma versão anterior 
Algo que funcionava parou de funcionar após as últimas alterações.
Preciso voltar à versão anterior do projeto.

para eu identificar qual versão estava funcionando.

Depois, me explique como voltar para essa versão sem perder
o histórico do projeto.

Quando pedir ajuda fora do Claude Code

O Claude Code resolve a grande maioria dos problemas de desenvolvimento. Mas existem situações onde buscar ajuda em outros lugares é mais eficiente: quando o mesmo erro persiste após três tentativas de correção diferentes, quando o problema envolve configurações específicas do Supabase ou do Vercel que estão fora do código, ou quando você precisa de uma segunda opinião sobre uma decisão de arquitetura.

A documentação oficial do Supabase (supabase.com/docs) e do Next.js (nextjs.org/docs) são referências confiáveis. A comunidade do Supabase no Discord também é ativa e responde perguntas com rapidez.

Regra de ouro

Nunca peça ao Claude Code para "fazer funcionar de qualquer jeito". Soluções improvisadas criam problemas maiores mais tarde. Prefira entender a causa do erro e corrigi-la corretamente, mesmo que leve mais tempo.

O caminho de agora em diante

Se você chegou até aqui e seguiu os passos, você tem um SaaS publicado na internet. Não um protótipo. Não uma ideia. Um produto real, com banco de dados, autenticação, funcionalidades e uma landing page para apresentá-lo.

O que você construiu

Você passou por um processo completo: encontrou um nicho com um problema real, definiu um produto com escopo claro, planejou a construção com um CLAUDE.md bem estruturado, configurou uma stack técnica profissional com Next.js, Supabase e Vercel, construiu as funcionalidades principais, publicou o produto na internet e criou uma landing page para apresentá-lo.

Cada uma dessas etapas tem valor independente. O processo de encontrar um nicho e definir um produto é aplicável a qualquer projeto, não apenas a SaaS. O fluxo de construção com Claude Code é reutilizável para qualquer produto que você queira construir no futuro. O que você aprendeu aqui não se limita a este projeto.

O que vem depois do MVP

O MVP que você construiu é o ponto de partida, não o destino. O próximo passo é colocar o produto nas mãos de usuários reais e observar como eles o usam. Quais funcionalidades eles usam mais? Onde eles travam? O que eles pedem que não existe ainda?

Essas respostas vão guiar o desenvolvimento das próximas versões. Não construa funcionalidades baseadas em suposições. Construa baseado no que os usuários reais precisam, observado a partir do uso real do produto.

O Claude Code vai continuar sendo seu parceiro nessa evolução. Cada nova funcionalidade segue o mesmo processo: descrever com precisão, construir, revisar, ajustar. Com o CLAUDE.md atualizado e o histórico do projeto no Git, o contexto está sempre disponível para continuar de onde parou.

Como continuar evoluindo o produto

À medida que o produto cresce, o CLAUDE.md precisa crescer junto. Toda vez que você adicionar uma funcionalidade significativa, atualize o CLAUDE.md para refletir o novo estado do produto. Isso garante que o Claude Code sempre tenha o contexto correto, independentemente de quando você retomar o trabalho.

Mantenha o hábito de fazer commits frequentes no Git. Cada funcionalidade concluída merece um commit com uma mensagem descritiva. Isso cria um histórico claro do que foi construído e quando, e facilita voltar a versões anteriores se necessário.

O que este ebook não cobre

Este ebook cobriu o caminho do zero ao produto publicado. Existem etapas importantes que ficaram fora do escopo e que você vai precisar explorar quando chegar nelas.

Pagamentos: integrar um gateway de pagamento como Stripe ou AbacatePay para cobrar pelos usuários do produto. Essa é a etapa que transforma o produto em um negócio com receita.

Marketing e aquisição: como encontrar os primeiros usuários, como apresentar o produto para o nicho que você escolheu, como crescer a base de usuários de forma sustentável.

Operações: monitoramento do produto em produção, alertas quando algo quebra, análise de uso para entender como os usuários interagem com o produto.

Cada um desses temas tem recursos específicos disponíveis. Quando você chegar nessas etapas, você já vai ter o produto funcionando e usuários reais para guiar as decisões. Esse é o melhor ponto de partida para qualquer uma delas.

O produto que você construiu hoje é a versão mais simples do que ele pode se tornar. O que importa é que ele existe, está publicado e resolve um problema real para pessoas reais.
